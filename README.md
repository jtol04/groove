## Project Abstract

Music streaming platforms have built-in music recommendation algorithms that recommend users songs based on playlists they've created and songs they've listened to. Groove is an app that takes inspiration from this and learns a user's music preferences through a simple and interactive swipe interface (similar to Tinder) and recommends new songs even before they listen. 

Groove is built around a machine learning classifier trained on Spotify audio features (danceability, energy, valence, tempo, acousticness, etc.) from the Spotify Tracks Dataset. Users swipe right or left on short song previews, and the system builds a personalized taste profile. A trained model then scores unseen tracks and returns recommendations ranked by predicted enjoyment.

**Input:** User can swipe to like or dislike the songs, each linked to its Spotify audio features. 

**Output:** A personalized taste profile dashboard and a ranked list of recommended tracks. The ranking system will be stored through a priority queue. The AI-generated frontend displays the swipe UI, taste dashboard, and recommended tracks while the backend handles model inference, user session management, and data storage.

**Testing/Evalutation:** See whether recommendation list and recommendation % changes adapts based on the user swipes.

## Data Structures

- **Max-heap (priority queue)** — After the classifier scores all unseen tracks, `heapq.nlargest` extracts the top-N recommendations in O(n log k) time rather than sorting the full dataset (O(n log n)). This keeps `GET /recommendations` fast even against 114k tracks.
- **Hash map (dict)** — An in-memory dictionary maps each `track_id` to its preloaded audio feature vector. This avoids redundant DB reads during inference: the model gets O(1) feature lookup per candidate track.
- **Deque (double-ended queue)** — The cold-start candidate pool is managed as a `collections.deque`. Songs are served from the front (O(1) pop) and new batches are appended to the back, giving a clean FIFO queue for sequencing swipe cards before the model has enough data to personalize.
- **Stack (list used as stack)** — Each user session maintains a swipe history stack. Swipes are pushed on every decision and popped to support "undo last swipe," allowing the user to reverse a misclick without corrupting their taste profile.
